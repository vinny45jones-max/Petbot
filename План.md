# План деплоя на Railway

## Что уже подготовлено в проекте

- Добавлен файл `railway.json` с явной командой запуска `python bot.py`.
- Добавлен `.env.example` со списком переменных окружения для Railway.
- Бот переведён на переменную `BOT_DATA_DIR`, чтобы файлы состояния можно было хранить на Railway Volume, а не в корне проекта.
- В `.gitignore` добавлены `runtime/` и `*.pid`, чтобы локальные служебные файлы не мешали деплою.

## Что нужно сделать в Railway

1. Создать новый проект Railway и подключить репозиторий.
2. Создать один обычный service из этого репозитория.
3. Убедиться, что Railway использует конфиг из `railway.json`.
4. В переменные окружения добавить:
   - `TELEGRAM_BOT_TOKEN`
   - `OPENAI_API_KEY`
   - `TELEGRAM_CHANNEL_ID`
   - `TELEGRAM_SPECIAL_CHANNEL_ID`
   - `CONTACT_PHONE`
   - `CARD_NUMBER`
5. Если нужен постоянный счётчик питомцев и сохранение сессий между рестартами:
   - добавить Railway Volume
   - смонтировать его, например, в `/data`
   - выставить переменную `BOT_DATA_DIR=/data`
6. Запустить деплой.
7. Проверить логи сервиса:
   - должно появиться `Start polling`
   - не должно быть ошибок `TELEGRAM_BOT_TOKEN не задан` или `OPENAI_API_KEY не задан`

## Важные замечания

- Публичный домен для этого бота не нужен, потому что он работает через long polling, а не через webhook.
- Если не подключать Volume, то `latest_sessions.json` и `pet_counter.txt` будут сбрасываться после redeploy/restart.
- Одновременно должен работать только один экземпляр polling-бота, иначе Telegram начнёт отвечать `Conflict: terminated by other getUpdates request`.

## Что проверить после запуска

1. Отправить `/start` в Telegram.
2. Пройти сценарий до генерации фото и текста.
3. Проверить публикацию в канал.
4. Убедиться, что после рестарта Railway бот снова поднимается автоматически.
