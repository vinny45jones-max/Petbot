# Зомби dev-сервер на auto-increment порту даёт ложный «фикс не работает»

## Контекст

Чинил admin-500. Применил фикс, перезапустил `npm run dev`, проверил `curl localhost:3000/admin` → снова **500**. Чуть не списал фикс как нерабочий и не пошёл копать дальше по ложному следу.

На деле: на `:3000` висел **старый dev-сервер из прошлой сессии** (чужой PID, старый код). Мой новый сервер увидел занятый порт и молча поднялся на **`:3001`** — в баннере было `⚠ Port 3000 is in use by process NNNN, using available port 3001 instead`. Я проверял зомби, а не свой фикс. На `:3001` всё было зелёное.

## Суть

Dev-серверы (Next `next dev`, Vite) при занятом порту **не падают, а берут следующий свободный** и пишут об этом одной строкой в баннере. Если её проигнорировать, проверка бьёт в чужой/устаревший процесс → ложный отрицательный результат и дебаг несуществующей проблемы.

Правило проверки фикса:
1. **Читай баннер старта** — на каком порту сервер реально поднялся (`Local: http://localhost:PORT`). Бей curl'ом именно туда.
2. Логируй сервер в файл и смотри порт: `npm run dev > dev.log 2>&1 &` → в `dev.log` строка `using available port` = ты не на дефолтном порту.
3. **Убей зомби перед проверкой.** Windows: `netstat -ano | grep LISTENING | grep :3000` → `taskkill //PID <pid> //F`. Один канонический сервер на известном порту — меньше путаницы.
4. Сопоставляй наблюдаемое с правкой: если поведение в точности «как до фикса», первая гипотеза — **проверяешь не тот процесс/порт/сборку**, а уже потом «фикс неверен».

## Почему это важно

Ложный отрицательный при верификации дороже самого бага: уводит в дебаг места, которое уже починено. Цена — целый виток systematic-debugging по призраку. Один взгляд на порт в баннере экономит его. Перекликается с verification-before-completion: проверять надо именно тот артефакт, что собрал, а не оставшийся от прошлого прогона.

## Когда применять

- Любая ручная проверка фикса на localhost после рестарта dev-сервера.
- Особенно когда в сессии было несколько перезапусков/воркспейсов (легко оставить висящий node).
- Симптом «как было до фикса» (а не новая ошибка) — сильный признак неправильной цели проверки, а не неверного фикса.

## Примеры

```text
# Баннер — ЛЕГКО проскочить глазами:
⚠ Port 3000 is in use by process 7944, using available port 3001 instead.
   ▲ Next.js 15.4.11
   - Local:        http://localhost:3001      <-- проверять ЭТОТ порт

# Было (ложный вывод): curl localhost:3000/admin → 500  (зомби, старый код)
# Стало:               taskkill //PID 7944 //F ; curl localhost:3001/admin → 200
```
